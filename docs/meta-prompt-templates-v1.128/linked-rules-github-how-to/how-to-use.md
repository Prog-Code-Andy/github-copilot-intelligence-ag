Here's how to actually run this bundle to configure a new project's .github workflow environment.

## What the bundle is

It's a 3-step pipeline of meta-prompts plus 5 reference rule files. You don't "run" all 9 files yourself — you point an AI agent (in VS Code chat/agent mode) at the pipeline entry file, and it walks through the steps, reading the rule files only when it needs them.

```
00-README.md  → orientation (you don't need to attach this, it's for context)
01-wrapper-structure-initializer.md  → Step 1: scaffold .github/
02-universal-meta-prompt-builder.md  → Step 2: generate the actual agent/instructions/skills/prompts/hooks
03-build-linking-compiler.md         → Step 3: link everything + report
rules/*.rules.md                     → consulted automatically by Step 2, not run standalone
```

## Step-by-step execution

### Step 0 — Open the right project
Open the **target repository** (the new project you want to configure) in VS Code — not this `github-copilot-intelligence` repo. This bundle is meant to be applied *to* other projects.

### Step 1 — Scaffold the .github skeleton
In VS Code chat (Plan mode recommended first), attach and run:

```
#file:01-wrapper-structure-initializer.md

Initialize the .github structure for this repository.
```

This creates `.github/{agents,instructions,prompts,skills,hooks/scripts,workflows,ISSUE_TEMPLATE}/`, `.gitkeep` placeholders, `PULL_REQUEST_TEMPLATE.md`, `CODEOWNERS`, `dependabot.yml`, `copilot-instructions.md`, and root `AGENTS.md`. It won't create any actual agent/skill/prompt/hook files — just the skeleton. Review the plan, then approve.

### Step 2 — Generate the capability package
Attach the core builder and describe what you need:

```
#file:02-universal-meta-prompt-builder.md

Agent name or intended role: Python Builder
Requested operation: CREATE
User request: Create an agent that implements Python features following
Clean Architecture, writes tests, and enforces security standards.
Available references:
  docs/requirements.md
  docs/architecture.md
  (any existing agents/instructions/prompts you already have)
```

Use this same file for **any** component type — Python, security, architecture, QA, whatever agent you need — and for any operation: `CREATE | AUDIT | MODIFY | MODERNIZE | EXTEND`. It will:
1. Discover existing evidence (docs, code, existing .github files) instead of quizzing you.
2. Ask only genuinely blocking questions.
3. Present a proposal (what it will create/modify/move) — **approve before it writes files**.
4. On approval, generate the agent `.agent.md`, plus only the instructions/skills/prompts/hooks that are actually justified — consulting `rules/agents.rules.md`, `rules/instructions.rules.md`, `rules/skills.rules.md`, `rules/prompts.rules.md`, `rules/hooks.rules.md` automatically as it builds each type. You don't need to attach those rule files yourself; the builder file references them by relative path and the agent will read them when needed.

### Step 3 — Link everything together
After Step 2 finishes and files exist, run:

```
#file:03-build-linking-compiler.md

Link the components just generated for the Python Builder agent.
```

This scans the now-populated .github, adds documented cross-references (agent → instructions it uses, skill → agents that use it, etc.), and produces an architecture report. Approve its plan before it edits files.

## Practical tips

- **Repeat Steps 2–3 per agent.** Step 1 only needs to run once per repo (or when the skeleton is missing/incomplete). Steps 2 and 3 run every time you add or modify a capability (e.g., once for a Python agent, again later for a Security Reviewer agent).
- **Use Plan mode first**, especially for Steps 1–2, so you see the proposal before anything is written — both files have explicit approval gates.
- **Don't skip Step 3.** A capability package generated in Step 2 isn't "done" per the builder's own rules until the linking step has run.
- **Issue templates and workflows** aren't auto-generated by default — Step 2 only creates a GitHub Actions workflow or issue template when your request or discovered evidence justifies it (least-surprise, reuse-first).
- **You can attach multiple rule files manually** if you want to sanity-check the agent's output yourself, e.g. `#file:rules/hooks.rules.md` to verify a generated hook uses a real event name and not a fabricated one.
